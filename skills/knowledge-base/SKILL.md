# 知识库创建 Skill

为指定领域创建完整结构化知识库：先规划大纲，再逐篇生成笔记，最终产出可检索、可扩展的知识体系。

## 写入规范

本 Skill 使用 Obsidian 兼容的 frontmatter 标准和路径规则：
- 每篇笔记必须包含完整 frontmatter（见下方模板）
- 文件名使用中文或 `{编号}-{篇名}.md` 格式
- 篇与篇之间用 `[[双向链接]]` 连接

## 核心流程

```
用户提出领域需求
    ↓
1. 调研：WebSearch 了解领域全貌
    ↓
2. 规划：生成知识库大纲（目录结构 + 每篇概述）
    ↓
3. 确认：展示大纲给用户，获得确认或修改意见
    ↓
4. 批量生成：逐篇创建笔记文件
    ↓
5. 索引：创建 README.md 索引文件
    ↓
6. 汇总：报告完成情况
```

## 第一步：调研与规划

### 调研策略
1. WebSearch 搜索该领域的核心概念、框架、知识体系
2. 识别领域的「心智模型」——该领域专家通常用什么框架思考？
3. 确定知识库的边界——包含什么，不包含什么

### 大纲生成规则

**规模控制：**

| 领域复杂度 | 建议篇数 | 生成策略 |
|---|---|---|
| 小（单一概念） | 3-5 篇 | 顺序生成 |
| 中（方法论/技术栈） | 5-15 篇 | 分批并行（Agent） |
| 大（完整学科） | 15-30 篇 | 分批并行，按主题分组 |

**大纲格式：**

```markdown
# {领域名称} 知识库大纲

## 1. {主题一：基础层}
### 1.1 {篇名} — {一句话描述}
### 1.2 {篇名} — {一句话描述}

## 2. {主题二：核心层}
### 2.1 {篇名} — {一句话描述}

## 3. {主题三：应用层}
### 3.1 {篇名} — {一句话描述}

## 4. {主题四：进阶/前沿}
### 4.1 {篇名} — {一句话描述}
```

**组织原则：**
- 从基础到应用，从通用到具体
- 每篇3000-5000字，聚焦一个清晰的知识点
- 篇与篇之间用 `[[双向链接]]` 连接

## 第二步：用户确认

展示大纲后等待用户反馈：
- ✅ 确认 → 开始生成
- ✏️ 修改 → 调整大纲后重新确认
- ➕ 增加/删除篇目 → 更新大纲
- ❌ 取消 → 终止

**不要跳过确认步骤直接开始生成。**

## 第三步：批量生成

### 生成策略

**小规模（≤5篇）**：顺序生成，当前会话直接写文件。

**中大规模（>5篇）**：使用 Agent 工具并行生成：
1. 按主题分组（每组 2-5 篇）
2. 每组启动一个后台 Agent（`run_in_background: true`）
3. 每个 Agent 收到明确的：文件路径、内容要求、字数、frontmatter 模板
4. 所有 Agent 完成后汇总检查

### 单篇笔记模板

每篇笔记必须包含：

```yaml
---
title: "{{title}}"
agent: "{{agent_id}}"
type: knowledge
date: {{date}}
status: draft
tags: [{{领域名}}, {{子领域}}, {{关键词1}}, {{关键词2}}]
domain: {{领域}}
depth: 1
---
```

**正文结构：**
1. **一句话定义**（开篇 1-2 句话说清楚这是什么）
2. **核心概念**（拆解关键要素）
3. **运作机制**（它是怎么工作的）
4. **与其他概念的关联**（`[[双向链接]]`）
5. **应用场景**（在什么情况下使用）
6. **常见误区**（容易犯的错）
7. **延伸阅读/参考资料**

### 质量标准
- 每篇 3000-5000 字（中文）
- 引用至少 2 个不同来源的信息
- 标注关键概念的双向链接
- 用 Mermaid 图表表达复杂关系（推荐但非必须）

## 第四步：创建索引

所有笔记生成完毕后，创建 `README.md` 索引文件：

```markdown
---
title: "{领域名称} 知识库"
type: system-doc
date: {{date}}
tags: [知识库, {领域名}]
status: active
---

# {领域名称} 知识库

> 创建日期：{{date}} | 篇数：{{总数}} | 创建者：{{agent}}

## 目录

### 1. 基础层
- [[篇名1]] — 一句话描述
- [[篇名2]] — 一句话描述

### 2. 核心层
...

## 学习路径

推荐阅读顺序：...

## 更新日志

| 日期 | 操作 | 说明 |
|---|---|---|
| {{date}} | 创建 | 初始版本，共 {{总数}} 篇 |
```

## 保存位置

**启动时必须询问用户：** "知识库保存在哪个目录？"

如果用户没有指定路径，建议默认位置：`./{领域名}/`（当前工作目录下）。

**目录结构：**
- README.md → `{根目录}/{领域名}/README.md`
- 笔记文件 → `{根目录}/{领域名}/`
- 如有子分类 → `{根目录}/{领域名}/{子分类}/`

**命名规则：** `{编号}-{篇名}.md`（如 `01-核心概念.md`）或直接 `{篇名}.md`

## 完成报告

生成完毕后，向用户汇报：

```
✅ {领域名称} 知识库创建完成

📁 位置：2-Projects/{领域名}/
📊 规模：{N} 篇笔记 + 1 个索引
⏱️ 耗时：{时间}
🔗 关键链接：{N} 个双向链接

### 目录结构
1. 基础层（{n}篇）
2. 核心层（{n}篇）
3. 应用层（{n}篇）
...

💡 建议：按 01 → 02 → ... 的顺序阅读
```

## 核心原则

### 结构化优先
1. 知识库是**体系**，不是文章堆砌
2. 每篇笔记是体系中的一个节点，节点之间有连接
3. 大纲决定了体系的质量，值得花时间打磨

### 可扩展性
1. 知识库创建后可以随时添加新笔记
2. 用 `status: draft` 标记新笔记，后续可更新为 `active`
3. 双向链接使得知识库随使用自然生长

### 质量控制
1. 多源验证，避免单一来源偏见
2. 客观事实与主观理解分开
3. 标注信息来源，便于后续溯源

## 注意事项

1. **先调研再规划**：不要凭空编造大纲，先了解领域全貌
2. **用户确认后才开始**：大纲确认是必须步骤
3. **并行生成注意并发限制**：后台 Agent 同时最多 8 个
4. **frontmatter 必须完整**：agent/type/date/status/tags 缺一不可
5. **双向链接使用标准名**：`[[首字母大写的概念名]]`
